# 00. システム概要

## 1. システム名称

保険契約・加入者管理システム（PoC）

## 2. システムの目的

本システムは、以下の業務を一貫して管理する社内向け Web アプリケーションである。

- 加入者（契約者・被保険者）の登録・管理
- 保険申込の受付
- 契約の審査・承認
- 契約変更（住所変更、連絡先変更、保障内容変更、解約）
- 給付請求の受付・審査・支払管理
- 監査ログ、マスタ管理、ユーザー管理

本システムは本番業務利用ではなく、以下を目的とした **PoC 用アプリケーション** として構築する。

- AI を利用した E2E テストシナリオ生成の検証
- AI を利用した自動テストコード生成の検証
- 状態遷移、権限制御、検索、一覧、明細入力、添付ファイル有無を含む業務アプリの再現
- React + Python(Lambda/API Gateway) + PostgreSQL 構成での実装性確認

## 3. 対象業務範囲

### 3.1 対象に含むもの

- 認証・認可
- 加入者管理
- 契約申込・契約管理
- 契約審査
- 契約変更申請
- 解約申請
- 給付請求受付
- 給付請求審査
- 支払確定
- 管理者機能（ユーザー、ロール、保険商品マスタ）
- 監査ログ参照

### 3.2 対象に含まないもの

PoC のため、以下は対象外とする。

- 実際の保険料計算の高度な数理ロジック
- 外部システム連携（収納、会計、メール配信、本人確認）
- PDF 帳票生成の高度なレイアウト制御
- 複雑なワークフローエンジン
- リアルタイム通知
- S3 連携など実ファイル格納の詳細実装  
  ※添付ファイルはメタデータ管理のみでも可

## 4. 想定ユーザー

- 申込担当者
- 審査担当者
- 支払担当者
- 管理者

## 5. 想定保険商品

PoC では以下 3 商品に限定する。

1. 医療保険
2. がん保険
3. 傷害保険

## 6. 全体アーキテクチャ

### 6.1 フロントエンド

- React
- SPA 構成
- 画面遷移は React Router を想定
- 認証後にロールに応じたメニューを表示

### 6.2 バックエンド

- API Gateway + AWS Lambda(Python)
- REST 風 API
- 業務単位で Lambda を分けても、リソース単位で分けてもよい
- 入力バリデーションは API 層とフロント双方で実施

### 6.3 DB

- PostgreSQL
- 論理削除フラグは極力使わず、ステータスと監査履歴で状態管理する
- ただしユーザー、マスタなど一部は無効フラグを持たせる

## 7. 設計方針

- PoC であっても、画面・API・DB の責務を分離する
- 画面仕様は実装可能な粒度にする
- 状態遷移は明示的に管理する
- 一覧検索・詳細・登録・更新のパターンを揃える
- E2E テストを想定し、画面要素に安定した識別子（data-testid 等）を付与しやすい構造とする

## 8. 非機能の前提

- 同時接続数は少数を想定
- レスポンスは PoC として実用的なレベルであればよい
- 監査性重視のため、主要操作は監査ログへ記録する
- 個人情報を扱う想定のため、画面上の項目名・データモデルは業務システムらしい整合性を保つ
