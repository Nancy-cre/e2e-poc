# 40. E2E テスト PoC 方針

## 1. PoC の目的

本システムは、以下の AI 活用 PoC に適した題材として設計している。

1. 業務仕様書から E2E テストシナリオを AI に生成させる
2. 画面仕様・API 仕様・DB 定義を参照してテストケースを AI に補完させる
3. Playwright 等の自動テストコードを AI に生成させる
4. 失敗結果をもとに AI に原因分析させる

## 2. 本システムが PoC に向く理由

- 申込担当者 → 審査担当者 → 支払担当者 → 管理者という複数ロールがある
- 一覧、詳細、登録、編集、承認、差戻し、却下、支払確定という多様な操作がある
- ステータス遷移が明確
- 検索条件、入力バリデーション、権限制御がある
- 契約申込、変更申請、解約申請、給付請求という複数シナリオを持つ

## 3. 推奨 E2E シナリオ群

### 3.1 契約申込正常系

1. 申込担当者ログイン
2. 加入者新規登録
3. 契約申込作成
4. 申請
5. 審査担当者ログイン
6. 審査承認
7. 契約一覧で ACTIVE 契約を確認

### 3.2 契約申込差戻し系

1. 申込担当者が申込作成
2. 審査担当者が差戻し
3. 申込担当者が差戻し理由を確認
4. 修正して再申請
5. 審査担当者が承認

### 3.3 契約変更系

1. ACTIVE 契約を開く
2. 住所変更申請を作成
3. 審査承認
4. 契約詳細で住所変更反映を確認

### 3.4 解約系

1. ACTIVE 契約から解約申請
2. 審査担当者が承認
3. 契約ステータスが CANCELLED に変わることを確認

### 3.5 給付請求から支払まで

1. ACTIVE 契約から給付請求作成
2. 審査担当者が承認
3. 支払担当者が支払確定
4. 請求ステータスが PAID になることを確認

### 3.6 権限制御

- 申込担当者が管理画面 URL を直接叩いて 403 または遷移抑止される
- 支払担当者に審査承認ボタンが表示されない
- 管理者のみ監査ログ参照できる

## 4. AI に渡しやすい情報

以下の情報が揃っているため、AI でのシナリオ生成がしやすい。

- 画面一覧
- 各画面の入力項目
- ステータス一覧
- 業務フロー
- API 一覧
- テーブル定義

## 5. テストデータ戦略

### 5.1 初期データ

- ロール 4 種
- テストユーザー 4 名
- 保険商品 3 件
- 保険プラン 6 件程度
- サンプルお知らせ数件

### 5.2 シナリオデータ

- 契約者と被保険者が同一のケース
- 契約者と被保険者が別人のケース
- 差戻し済の申込
- 支払待ち給付請求
- 解約済契約

## 6. 画面実装時の推奨ルール

E2E 安定性のため、以下を推奨する。

- ボタンに `data-testid` を付与
- 一覧テーブル行に安定した属性を付与
- 日付入力コンポーネントを統一
- 非同期ロード中のローディング表示を統一
- 成功 / 失敗トースト文言を定型化
- 確認ダイアログの UI を統一

## 7. Claude Code に追加で渡すと良い指示

- React コンポーネント構成
- 画面レイアウト共通化方針
- API クライアントの抽象化方針
- Lambda ディレクトリ構成
- SQLAlchemy / psycopg 等の利用方針
- seed データ作成方針
- Playwright テストコード生成ルール

## 8. 実装優先順位

### フェーズ1
- 認証
- 加入者管理
- 契約申込
- 審査
- 契約一覧 / 詳細

### フェーズ2
- 契約変更
- 解約
- 給付請求
- 支払処理

### フェーズ3
- 管理機能
- 監査ログ
- テストデータ整備

## 9. PoC 成果物として期待するもの

- 仕様書から実装された業務アプリ
- E2E テストシナリオ一覧
- 自動テストコード
- AI による失敗分析結果
- 仕様変更時の差分生成検証
